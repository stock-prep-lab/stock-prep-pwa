# AGENTS.md

このリポジトリで作業するエージェントは、以下のルールに必ず従うこと。

## 基本方針

- すべての PR は日本語で作成すること。
- 実装は Slice 単位で進めること。
- 一度に大きな機能をまとめて実装しないこと。
- レビュー可能な最小単位で PR を作成すること。
- 仕様に書かれていない大きな拡張を勝手に追加しないこと。
- 将来拡張の余地を残しつつ、まずは MVP を優先すること。

## 作業時の優先順位

1. `docs/plan/slices.md`
2. `docs/requirements/*`
3. `docs/design/*`

上記の順に従い、対象 Slice の範囲を超えた作業は行わないこと。

## ブランチと PR のルール

- 作業は `main` ブランチへ直接 push せず、必ず作業ブランチで行うこと
- 各 Slice の実装が完了したら、必ず Pull Request を作成すること
- Pull Request は 1 Slice に対応する 1 PR を原則とすること
- PR を作成せずに作業完了とみなさないこと
- PR タイトルは対象 Slice と内容がわかるように簡潔に書くこと

### PR タイトル例

- `Slice 0: 開発基盤を構築`
- `Slice 2: ホーム画面の静的 UI を追加`
- `Slice 9: IndexedDB 基盤を追加`

## PR の粒度

- 原則として 1 PR = 1 Slice
- 1 PR で複数画面をまたぐ大規模実装をしない
- UI とロジックの両方を大きく同時変更しない
- 画面を作る Slice と、データ接続を行う Slice は可能な限り分ける
- レビュー担当者が 1 回で追える量に抑えること

## 実装ルール

- 型を先に定義する
- 共通ロジックは `packages/domain` または `packages/shared` に寄せる
- 画面固有ロジックを直接コンポーネントへべた書きしすぎない
- ダミーデータで組む Slice と、本物のデータをつなぐ Slice を分ける
- 大きな状態管理導入は必要最小限にする
- 未確定要件は仮実装せず TODO として明示する
- 将来用の過剰な抽象化は入れない

## テスト追加ルール

- 各 Slice では、その Slice に見合うテストまたは確認を必ず追加すること
- ロジック追加時はユニットテストを優先すること
- UI 追加時は最低限の表示確認観点を PR に記載すること
- 主要導線が完成した段階で E2E テストを追加すること
- E2E テストは画面骨組みが固まる前に無理に書かないこと
- `lint`, `typecheck`, `build` を最低限の品質ゲートとすること
- テストを追加しない場合は、その理由を PR に明記すること
- `docs/plan/slices.md` に E2E 対象 Slice が定義されている場合、その Slice では E2E テストの追加または更新を必須とすること
- E2E 対象 Slice で E2E を追加しない場合は、その理由を PR に明記すること

## テストと品質

- 変更に応じて `lint`, `typecheck`, `test` を通すこと
- UI を変更した場合は最低限の確認観点を PR に書くこと
- 壊しやすいロジックにはユニットテストを追加すること
- build が落ちる状態で PR を作らないこと

## レスポンシブ対応

- UI を変更した場合は、モバイル幅とデスクトップ幅の両方で崩れがないことを確認すること
- 完了条件には、主要画面がモバイル幅とデスクトップ幅の両方で成立していることを含めること
- レスポンシブ対応は「見える」だけでなく、主要操作が無理なく行えることを確認すること
- 不要な横スクロールを前提にしないこと
- 主要ボタンや主要情報が画面外に隠れないこと

## PR に必ず含めること

- 目的
- この PR の対象
- この PR の非対象
- 変更した文書
- 変更理由
- 運用上の影響
- 技術的または設計上の学び
- テスト内容
- 画面確認結果（モバイル幅 / デスクトップ幅）
- 次の Slice への接続

## 禁止事項

- 一気に完成形まで作ること
- 仕様にない画面を大量追加すること
- 将来用の複雑な抽象化を入れすぎること
- レビュー困難な巨大 PR を作ること
- 対象 Slice 外の機能をまとめて入れること
